## Frameworks de compliance obligatoires

### 🇫🇷 Fiscalité - France 2026

#### Factur-X (EN 16931)

```yaml
Obligation:
  Tous invoices émis dès 2026 doivent être Factur-X native
  Format: PDF/A-3 embedding XML metadata (UBL-Invoice)
  Signature: Avancée (XAdES ou équivalent)
  Archivage: Immuable 20+ ans (NF Z 42-013)

Métadonnées requises:
  - Invoice number + date
  - Seller / Buyer TAX IDs (SIRET/TVA)
  - Line items + VAT breakdown
  - Payment terms
  - Signature timestamp (horloge serveur certifiée)

Validation:
  Schéma Zod obligatoire pré-sérialisation PDF
  Test E2E: Parse PDF → Extract XML → Validate UBL
```

#### NF525 (Imputabilité & Archivage)

```yaml
Standard Français:
  Tous docu comptables doivent être imputables = tracer qui a créé/modifié
  Hash chain: SHA-256 de chaque document + timestamp monotone
  Archivage: Stockage immuable 6 ans minimum (audit-proof)

Implementation:
  - Chaque facture: hash(document + previous_hash + timestamp_serveur)
  - Backup: Chaîne hash dans Restic + validation monthly
  - Audit: Générer "Certificate of Digitalization" (preuve légale)
```

### 🇪🇺 RGPD - Donnée Personnelle

#### Data Inventory & Mapping

```yaml
Données personnelles identifiées dans micro-gestion-facile:
  Client:
    - Nom, Adresse: PII (need consent)
    - Email: PII (need consent)
    - Phone: PII (need consent)

  User (Admin/Accountant):
    - Auth credentials: SENSITIVE (encrypt at rest + TLS transit)
    - Activity logs: LEGITIMATE_INTEREST (retention: 2 ans)

  Transactions:
    - Invoice amounts: NON-PII (public-safe)
    - Payment timestamps: NON-PII (public-safe)
    - Bank details: SENSITIVE (never log, hash only for reconcile)

Storage:
  PII: IndexedDB (local) + Restic backup (encrypted AES-256)
  Bank: NEVER stored plaintext → hash + vault for compare only
  Logs: Scrubbed (No PII in console logs / server logs)
```

#### Droit des Personnes

```yaml
Data Subject Rights (Articles 12-22 RGPD):

  1. Right to access:
     - User can export all personal data JSON (Button: "Download My Data")
     - Include: Invoices, auth logs, preferences

  2. Right to rectification:
     - User can edit own PII (name, email, address)
     - Audit: track old_value → new_value + timestamp

  3. Right to erasure ("Right to be forgotten"):
     - User lifetime account deletion = nullify PII fields
     - Invoices: keep (legal obligation NF525) but anonymize
     - Backups: Retain until retention policy expires

  4. Right to object:
     - Telemetry opt-out: GDPR cookies consent required
     - Marketing: Email unsubscribe + preference flag

  5. Data portability:
     - Export format: JSON or Factur-X PDF
     - Standard format (not locked to app)

  6. Restrict processing:
     - "Limit to invoicing only" flag
     - Exclude from analytics / IA training
```

#### DPA (Data Processing Agreement)

```yaml
Si tiers integration (e.g., Stripe, Sentry, Analytics):

  Checklist PRE-integration:
    [ ] DPA signé avant activation
    [ ] Localisation serveur confirmée (EU-only if RGPD critical)
    [ ] Sous-traitants listés et approuvés
    [ ] Opt-out possible (no lock-in)

  Audit clause:
    - Tiers doit permettre audit indépendant
    - Restriction géographique (data residency)
    - Notification incidents < 72h per Art. 34
```

### 🔐 Privacy-by-Design (PbD)

```yaml
Principes PbD appliqués à micro-gestion-facile:

1. Data minimization:
  - Collecte: Minimum requis pour facture valide
  - Exemple: Collecter zip code ≠ nécessaire → déduire de SIRET client

2. Local-first storage:
  - Défaut: IndexedDB client-side
  - Sync optionel: POST vers serveur si utilisateur accepte
  - Jamais: Env de development = données de prod

3. Pseudonymization:
  - Backups: Hash email pour analytics (non-reversible)
  - Logs applicatif: Anonymiser PII avant persist

4. Encryption:
  - Transit: TLS 1.3 mandatory
  - At rest: AES-256-GCM pour Restic backup
  - Keys: Vault-managed, rotated 180j

5. Consent management:
  - Implicit refusal: Chaque feature PII require consent
  - Widget: Afficher consentements actifs (user agency)
  - Audit: Log every consent acceptance/revocation
```

## Audit Readiness Checklist

### Pre-audit (2 semaines avant inspection)

- [ ] **Factur-X validation**: Script E2E qui génère 100 factures teste parsing
- [ ] **Data inventory**: Spreadsheet PII / Non-PII / SENSITIVE + justif stockage
- [ ] **Consent records**: Exporter audit logs consentements (RGPD Art. 12)
- [ ] **Archivage NF525**: Vérifier hash chain intégrité 100% (monthly check log)
- [ ] **DPA check**: Tous tiers (Stripe, hosting, analytics) ont DPA signé
- [ ] **Encryption audit**: Certification TLS, Restic key management
- [ ] **Incident log**: Réview tout incident < 1 an (breaches, exposures)
- [ ] **Backup audit**: Restore complet depuis oldest backup (prove NF525 works)
- [ ] **Code audit**: Zero PII hardcoded, zero credentials in git history

### During audit (Inspection)

- [ ] Certificate of Digitalization (Factur-X validator report)
- [ ] Hash chain integrity proof (monthly audit export)
- [ ] Data inventory + DPA folder (organized, accessible)
- [ ] Consent audit trail (all acceptance/rejection records)
- [ ] Incident response log (breaches + remediation)
- [ ] Third-party audit (tiers certifications SOC 2 / ISO 27001)

### Post-audit (Remediation)

- [ ] Corrective actions logged + deadline tracked
- [ ] Evidence collection para chaque finding
- [ ] Re-audit scheduled (30-60 days)

- Audit: Code review focus RGPD/NF525 checkpoints

- Daily: Backup integrity check (hash chain validation)
- Weekly: Consent audit (expiring, revoked)
- Monthly: Drill → Restore depuis oldest backup, re-validate NF525
- Quarterly: External audit DPA compliance

### 📋 Incident Response

- Incident: Breach assessment < 24h
- Notification: CNIL reporting <= 72h if required
- Root cause: Document in audit log (prevent recurrence)
- Evidence: Preserve for auditors/legal

## Traits distinctifs

✅ **Priorités**:

- Compliance > Feature velocity (tech debt ok si audit-clean)
- Audit trail > UX friction (paper trail > happy user)
- Privacy-by-design > Retrofit compliance (late fixes cost $$)
- Consent explicit > Implicit opt-in (trust matters)

❌ **À éviter**:

- PII in logs, metrics, error messages
- Tiers sans DPA signée (legal liability)
- Undocumented data flows (audit nightmare)
- Consent dark patterns (dark mode = non-compliance)
