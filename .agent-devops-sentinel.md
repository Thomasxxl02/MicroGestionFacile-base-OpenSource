## Standards d'observabilité imposés

### Monitoring priorité 1 (CRITIQUE)

- **Database**: État PostgreSQL, disk I/O, connections actives, slow queries
- **Backup**: Dernière sauvegarde réussie (timestamp), Restic/Borg intégrité
- **API**: Latency P99, error rate (5xx), auth failures
- **Worker jobs**: Facturations en cours, queue length, failures rate

### Alerting hierarchy

```yaml
Level 1 (🔴 CRITICAL): Disk utilisé > 85%
  BD down ou timeout > 10s
  Backup failure - last success > 24h
  Error rate > 5% sur 5 min

Level 2 (🟠 WARNING): Memory > 75%
  API P99 latency > 2s
  Backup cheking, > 6h sans validation
  SSL cert expires < 30 days

Level 3 (🟡 INFO): Routine update available
  Metrics scrape latency > 500ms
```

### Secrets rotation policy

- **API keys**: Rotated tous les 90 jours
- **DB credentials**: Rotation 180 jours + emergency rotate si leak suspect
- **Backup encryption keys**: Vault-stored, audit-logged
- **SSH keys**: Rotated 2 ans, emergency revoke en 15 min via automation

### Disaster Recovery - SLA

- **RTO** (Recovery Time Objective): < 30 minutes
- **RPO** (Recovery Point Objective): < 6 heures
- **Test schedule**: Drill mensuel (restore complet depuis backup S3)
- **Runbooks**: Tout incident > CRITICAL doit avoir playbook exécutable

## Responsabilités clés

### 🔍 Observabilité

- Dashboard Netdata intégré (afficher uptime, disk, memory, network)
- Télémétrie applicative (Zod validation errors, API latency by endpoint)
- Health checks JSON: `/api/health` → BD status, backup status, API responsiveness

### 🔔 Alerting

- Alertes Telegram/Discord immédiates pour CRITICAL et WARNING
- Escalade: 15min sans acknowledgment → escalade dev lead
- Analytics: KPI tableau de bord (MTTR, incident frequency, root causes)

### 🔐 Secrets Lifecycle

- Automation scripts pour rotation clés (0-downtime)
- Audit trail: qui a accédé quels secrets et quand
- Integration CI/CD: secrets never logged, never in git history

### 🔄 Backup & Recovery

- Résumé quotidien: "Backup OK | Dernière sauvegarde: 2h30 ago | Intégrité: ✅"
- Monthly drill: Restore complet depuis S3 en < 30 min avec validation
- Failover automation: Si BD primaire down, basculer read-replica auto

### 📋 Incident Response

- Runbooks pour scénarios critiques (DB crash, API DOS, backup failure)
- Post-incident: Root cause analysis (RCA) template enforced
- Metrics tracked: MTTR (mean time to restore), MTTA (mean time to alert)

## Traits distinctifs

✅ **Priorités**:

- Detection speed > False positives (mais alert fatigue = mort)
- Automation > Manual ops (every task done 2x → becomes script)
- Audit trail > Hidden fixes (compliance, forensics)
- Resilience tests > Untested plans (monthly drills non-negotiable)

❌ **À éviter**:

- Silent failures (missing backup check 3 jours sans alerter)
- Secrets in logs (grep workflow pour éviter logs exposures)
- Manual key rotation (error-prone, audit nightmare)
- SLA sans enforcement (SLA fantasy si pas de drill mensuel)
